# 经验三级进化（Lessons Evolve）

> 核心理念：把一个坑踩一次写成 **lesson**，踩两次归纳为 **pattern**，踩三次晋升为 **instinct**。
> 每次晋升需要**人工确认**，防止错误经验扩散。
> 灵感来源：AI Coding Harness 工程化实践中的经验自动晋升机制。

## 设计哲学

团队/个人反复踩同一个坑，每次都“记住教训了”，但换个项目又忘了。memory 太容易淹没在碎片信息里，skill 又太正式。

用三级模型按“重复次数”自动逐级提升经验的固化程度：

```text
一次教训 -> 记忆记住就好
二次重复 -> 值得写成规则了
三次验证 -> 应该变成默认行为
```

跟 darwin-skill 的区别：

| 维度 | darwin-skill | lessons-evolve |
|------|-------------|---------------------|
| 方向 | 自上而下优化已有 SKILL.md 质量 | 自下而上从执行轨迹中提炼新规则 |
| 输入 | 已有 skill 文本 | 实际问题/踩坑/修正记录 |
| 输出 | 更好的 SKILL.md | 新 memory / 新 skill / 流程规则 |
| 触发 | 用户主动调用 | 结项、复盘、周报、故障恢复后自动触发 |

## 环境依赖与降级策略

本 skill 的正常运行依赖以下能力。执行前须先检查可用性，不可用时自动降级：

| 能力 | 场景 | 不可用时的降级行为 |
|------|------|-------------------|
| `memory` 或当前 Agent 的长期记忆能力 | 写 lesson 记录、存拒绝标记 | 降级为写到当前项目 `.lessons-evolve/lessons.yaml`，任务结束时提示用户该文件位置 |
| `session_search` 或当前 Agent 的历史检索能力 | 搜索历史同类问题 | 降级为搜索当前项目 `.lessons-evolve/lessons.yaml`，同时在回复中提示用户“无法自动查完整历史，请确认是否遇到过类似问题” |
| 当前 Agent 的 skill/rule 管理能力 | 创建 pattern 级别规则 | 降级为输出完整规则文本，并提示用户按当前 Agent 的方式保存为 skill、rule 或 AGENTS.md 片段 |
| 文件搜索/读取能力 | 搜索 lesson 文件 | 降级为检查当前项目是否存在 `.lessons-evolve/lessons.yaml`，有则读取文件 |

**环境检查步骤**（Step 0，在任何操作前执行）：

```text
1. 检查当前 Agent 是否提供长期记忆能力 -> 记录为 has_memory
2. 检查当前 Agent 是否提供历史检索能力 -> 记录为 has_history_search
3. 检查当前 Agent 是否提供文件搜索/读取能力 -> 记录为 has_file_access
4. 根据可用性选择对应的数据通道（见下文“证据仓库”）
```

## 证据仓库

lesson 证据不再仅靠 memory 一条记录，而是**结构化存储**，包含晋升所需的全部字段。

### 数据存储通道

优先顺序（按可用环境自动选择）：

| 优先级 | 存储位置 | 适用条件 |
|--------|---------|----------|
| 1 | 当前 Agent 的长期记忆能力 | `has_memory=true` |
| 2 | 当前项目 `.lessons-evolve/lessons.yaml` | `has_file_access=true`，文件系统可用 |
| 3 | 回复正文 | 数据通道都不可用时，在回复中输出，让用户自行保存 |

### lesson 记录模板

```yaml
- lesson_id: "ev-20260615-001"             # 格式：ev-YYYYMMDD-序号
  title: "一句话描述"                       # 示例：mvn -am 在 Mac + JDK17 下卡死
  symptom: "具体的现象/报错信息"             # 示例：mvn -am 执行后无响应
  root_cause: "根因"                       # 示例：-am 触发全量依赖解析
  solution: "修复步骤"                      # 示例：改用 mvn -pl <module> -am
  tags: ["build", "maven", "mac"]          # 搜索关键词
  source_session: "2026-06-15-maven-setup" # 来源会话标识
  source_file: "~/project/pom.xml"         # 来源文件，可选
  created_at: "2026-06-15T10:30:00Z"
  promotion_count: 0                       # 晋升拒绝计数
  last_hit_at: null                        # 上次被命中的时间
  status: "active"                         # active | user_rejected | superseded | reverted
```

### 数据查找方式

| 能力可用 | 查找方式 |
|----------|---------|
| 有历史检索能力 | 用关键词搜索过往 session 中的同类问题，并搜索长期记忆中的 lesson 记录 |
| 只有文件访问能力 | 搜索当前项目 `.lessons-evolve/lessons.yaml` 的 tag、title、root_cause |
| 什么搜索都不可用 | 提示用户手动确认是否以前遇到过，并建议先让用户搜索或回忆 |

## 晋升证据模型

### 晋升条件（精确版）

| 当前级别 | 晋升到 | 条件 | 证据要求 |
|----------|--------|------|---------|
| 无 | lesson | 首次遇到非显然问题 | 一条 lesson 记录写入证据仓库 |
| lesson | pattern | 第二次命中：同一 lesson 的 symptom、root_cause、solution 任一匹配 | session_search 或文件搜索命中至少 1 条 lesson，且 `promotion_count < 2` |
| pattern | instinct | 第三次命中：同一 pattern 被验证有效至少 2 次 | session_search 或文件搜索命中至少 2 条 lesson，且 `promotion_count < 2` |
| 任意 | 暂停提醒 | 用户连续拒绝 2 次同一 lesson 升 pattern | `promotion_count >= 2` 的 lesson 不再提示晋升 |

### 晋升计数规则

- lesson 记录含 `promotion_count` 字段。
- 每次用户拒绝晋升：`promotion_count += 1`。
- `promotion_count >= 2`：不再提醒晋升，标记 `status: user_rejected`。
- 晋升成功（lesson -> pattern 或 pattern -> instinct）：新建记录，旧记录标记 `status: superseded`。
- 晋升判断依据：**当前环境中最准确的可用搜索方式**的命中数，不强求 `session_search`。

## 三级模型

### 第一级：lesson（教训记录）

| 属性 | 说明 |
|------|------|
| 触发条件 | 第一次遇到某个坑/问题，详见“触发时机”限定 |
| 存储形式 | 按“数据存储通道”优先级写入 lesson 记录 |
| 作用范围 | 当前项目的下次会话，依赖 memory 或 lesson 文件加载 |
| 晋升条件 | 第二次在不同项目/不同场景再次遇到同类问题 |
| 行为 | 写入 lesson 记录，然后输出通知模板告知用户 |

lesson 写入后必须通知用户：

```text
已记录经验教训
- 问题：{title}
- 根因：{root_cause}
- 存储位置：{长期记忆 / .lessons-evolve/lessons.yaml / 回复正文}
- 证据 ID：{lesson_id}
- 下次遇到同类问题时我会提醒你，看是否要固化为规则。
```

### 第二级：pattern（模式规则）

| 属性 | 说明 |
|------|------|
| 触发条件 | 第二次遇到同类问题，搜索命中已有 lesson |
| 存储形式 | 当前 Agent 支持的 skill、rule 或 AGENTS.md 规则段；不可用时降级输出文本 |
| 作用范围 | 同类任务的所有会话 |
| 晋升条件 | 第三次验证确认后 |
| 行为 | 展示给用户确认，用户确认则创建 skill，然后输出通知模板 |

pattern 记录格式（作为 skill 的 SKILL.md）：

```markdown
---
name: <rule-name>
description: 自动从 lessons-evolve 晋升的规则
---

# <规则名称>

## 问题

{问题描述}

## 根因

{根因分析}

## 约束

{具体的避免方法/规则}

## 验证

{怎么检查这条规则是否被遵守}

## 来源证据

- lesson 1: {lesson_id} ({created_at}) - {title}
- lesson 2: {lesson_id} ({created_at}) - {title}
```

### 第三级：instinct（固化本能）

| 属性 | 说明 |
|------|------|
| 触发条件 | 第三次验证后，确认是通用规则 |
| 存储形式 | 固化到关联流程技能的规则部分/checklist，或全局 memory 指令 |
| 作用范围 | 所有相关工作流 |
| 行为 | 展示给用户确认，用户确认则执行固化，然后输出通知模板 |

instinct 的表现形式：不只是一个文件，而是**行为改变**：

- 开发流程规则：加到发布 checklist 或质量门禁。
- 工具使用规则：加到配置文件的约束部分。
- 业务知识：加到长期 wiki 或 codemap。

## 使用流程

### 触发时机（严格限定）

本 skill **仅在以下场景**自动触发，避免过度沉淀：

| 时机 | 触发条件 | 推荐动作 |
|------|----------|----------|
| 本轮执行有非显然修正 | 本次修改或排查过程中出现了至少一次非显然修正，即不是打字错误、不在 TDD 红绿循环的正常范围、不是显而易见的问题 | 走完整评估流程 |
| 重复错误 | session 中明确出现“又遇到了”“上次也这样”“同样的错误”等表述 | 立即启动晋升检查 |
| 故障恢复后 | 线上问题排查完毕，涉及根因分析 | 记录 lesson，检查是否可晋升 |
| 用户主动提 | “老遇到这个问题”“总结一下近期的教训”“把经验沉淀成规则” | 立即启动 |
| 用户明确要求复盘 | 周报/月报/结项时用户说“总结一下吧”或“有哪些教训” | 走完整评估流程 |

**不触发**的场景：

- 正常开发流程中的普通编译报错，修正后不再出现。
- TDD 红绿循环中的预期失败。
- 显而易见的配置疏忽，例如拼写错误、漏传参数。
- 每次部署都有的常规操作步骤。

### 标准执行流程

```text
用户表达“又遇到了”或满足触发条件
    │
    ├── Step 0: 环境检查
    │      - 检查长期记忆能力是否可用
    │      - 检查历史检索能力是否可用
    │      - 检查文件搜索/读取能力是否可用
    │      - 确定本次使用哪个数据通道
    │
    ├── Step 1: 识别问题
    │      - 当前遇到的问题/修正是什么？
    │      - 明确问题描述（symptom）、根因（root_cause）、解决方式（solution）
    │      - 判断是否属于非显然修正；不是则跳过
    │
    ├── Step 2: 搜索历史（按可用能力选择方法）
    │      ├── 有历史检索能力 -> 用关键词搜索过往 session
    │      ├── 有文件访问能力 -> 搜索 .lessons-evolve/lessons.yaml
    │      ├── 两者都有 -> 同时搜索，取并集
    │      └── 都不可用 -> 提示用户手动确认
    │
    ├── Step 3: 判断晋升等级与处理
    │      │
    │      ├── 无命中 -> 首次遇到
    │      │      ├→ 生成 lesson_id 并写入证据仓库
    │      │      └→ 输出通知模板告知用户
    │      │
    │      ├── 命中1次，promotion_count < 2 -> 第二次遇到
    │      │      ├→ 问用户确认：“第二次遇到了，要创建个规则兜底吗？”
    │      │      ├→ 用户确认 -> 按当前 Agent 支持方式创建 pattern 规则
    │      │      │     └→ 输出通知模板
    │      │      ├→ 用户拒绝 -> promotion_count += 1
    │      │      │     └→ 输出通知模板（含未晋升原因）
    │      │      └→ promotion_count >= 2 -> 不提醒，仅输出“之前已确认不处理”
    │      │
    │      └── 命中2次+，promotion_count < 2 -> 第三次遇到
    │             ├→ 问用户确认：“第三次了，要不要加到默认行为里？”
    │             ├→ 用户确认 -> 固化到相关流程（instinct）
    │             │     └→ 输出通知模板
    │             ├→ 用户拒绝 -> promotion_count += 1
    │             │     └→ 输出通知模板
    │             └→ promotion_count >= 2 -> 不提醒
    │
    └── Step 4: 输出最终通知（见“输出契约”）
```

### 搜索命中判断

判断两次问题是否为“同类”是关键：

1. **症状相同**：相同的报错信息、异常类型。
2. **根因相同**：同一类底层原因。
3. **解决方案相同**：用了同样的命令/步骤修复。

满足任一即判定为同类。由 AI 做语义判断。

## 输出契约

### lesson 通知模板

```text
已记录经验教训
- 问题：{title}
- 根因：{root_cause}
- 证据 ID：{lesson_id}
- 存储位置：{长期记忆 / .lessons-evolve/lessons.yaml / 回复正文}
- 后续处理：下次再遇到同类问题我会提醒你，可以考虑固化为规则。
```

### pattern 创建通知模板

```text
已创建规则“{skill_name}”
- 来源证据：lesson {lesson_id_1}、lesson {lesson_id_2}
- 规则内容：{简短描述}
- 下次遇到同类问题自动应用此规则。
```

### pattern 拒绝通知模板

```text
已跳过“{title}”的规则创建
- 原因：用户确认不处理
- 拒绝计数：{promotion_count}/2
- 后续处理：{promotion_count >= 2 ? '本问题将不再提醒。如需重新启用，请主动说“重新考虑 {title} 的规则”' : '下次再遇到时我还会提醒。'}
```

### instinct 固化通知模板

```text
已固化“{title}”为默认行为
- 固化位置：{具体的文件/流程/checklist}
- 生效范围：{影响范围}
- 如需回退，请说“撤销 {title} 的固化”执行撤销流程。
```

### instinct 撤销通知模板

```text
已撤销“{title}”的固化
- 受影响位置：{受影响项}
- 当前级别：回退至 pattern（skill 保留，但不再作为默认行为自动执行）
```

### 无变化通知模板

```text
经验检查完毕，本次无新经验需要记录。
```

## 可逆设计

### 降级流程

| 操作 | 触发条件 | 步骤 |
|------|----------|------|
| instinct -> pattern | 用户说“撤销 {title} 的固化” | 1. 从固化位置移除规则；2. 保留对应的 pattern skill；3. 输出撤销通知模板 |
| pattern -> lesson | 用户说“移除 {title} 的规则” | 1. 按当前 Agent 的规则管理方式删除对应 skill/rule/AGENTS.md 片段；2. 原 lesson 证据保留；3. 输出删除通知 |
| 完全删除 | 用户说“彻底忘掉 {title}” | 1. 删除 skill/rule/AGENTS.md 片段（如有）；2. 从 `.lessons-evolve/lessons.yaml` 或长期记忆中删除对应 lesson 记录；3. 输出确认 |
| 重新激活 | 用户说“重新考虑 {title}” | 1. 清除 `promotion_count` 和 `status`；2. 后续命中会重新提示晋升 |

## 示例场景

### 示例1：开发流程 - mvn -am 卡死

```text
第一次（项目A）:
  -> 输出 lesson 通知
  -> 证据：{lesson_id: ev-20260615-001, symptom: "mvn -am 卡死", tags: [maven, build, mac]}

第二次（项目B，不同仓库）:
  -> 搜索命中 lesson ev-20260615-001（symptom 匹配）
  -> AI 展示：“第二次遇到 mvn -am 卡死了。要建个规则兜底吗？”
  -> 用户确认 -> 创建 skill "maven-build-safety"
  -> 输出 pattern 创建通知

第三次（项目C）:
  -> pattern skill 自动加载：跳过 mvn -am
  -> AI 展示：“已稳定拦了 3 次，要不要加到编译流程的检查项？”
  -> 用户确认 -> 固化到编译质量门
  -> 输出 instinct 固化通知
```

### 示例2：运维 - 磁盘告警

```text
memory -> lesson_id: ev-20260301-003
  -> "机器 A 日志没轮转导致磁盘满 -> 需要加 logrotate 配置"

两个月后机器 B 同样告警 ->
  session_search 命中 -> 晋升 pattern
  创建 skill "disk-monitor-checklist"
  规则：“新机器上线时检查 logrotate 是否配置”

再遇到第三台 -> 晋升 instinct
  固化到服务器初始化 SOP
```

### 示例3：用户连续拒绝（对抗偏差）

```text
第一次 lesson 写入 -> 正常

第二次命中 -> AI 问“要建规则吗？”
  用户说“不用了” -> promotion_count = 1
  输出拒绝通知模板

第三次命中 -> AI 又问“又遇到了，这次呢？”
  用户说“说了不用了” -> promotion_count = 2
  输出“之前已确认不处理” -> 不再提醒

后来用户改主意 -> “重新考虑 mvn 的问题”
  清除 promotion_count
  下次命中恢复正常提示
```

## 对抗偏差

1. **人工确认门槛**：lesson -> pattern 需要确认，pattern -> instinct 需要确认。一个一次性的巧合不值得固化。
2. **聚焦同一问题**：只对比“症状/根因/修复方式”相同的问题，不强行合并“看起来相似但本质不同”的问题。
3. **停止打扰**：如果用户连续拒绝两次某个 lesson 升 pattern，标记 `status: user_rejected`，不再提醒。用户可主动“重新考虑”清除标记。
4. **可逆设计**：instinct 级别也可以回退，固化后发现不适用，按降级流程执行即可。
5. **不触发场景明确列明**：普通编译报错、TDD 预期失败、显而易见的问题不进入流程。
6. **环境降级透明**：每次执行记录使用了哪种数据通道，让用户知道能力边界。

## 关键原则

1. **AI 自动检查，人工决定晋升**：AI 负责发现“这个问题第二次出现了”，但要不要创建规则由用户决定。
2. **一次确认，永久受益**：创建 pattern/instinct 后，后续会话自动受益，不再反复踩。
3. **存证据**：每次晋升保留 lesson_id 和来源摘要，方便追溯“为什么会有这条规则”。
4. **错误经验不扩散**：晋升失败（用户拒绝）后计数累加，不再重复提醒。
5. **不替代正常学习**：已知的、本来就是流程一部分的步骤不需要走经验进化。只处理意外踩坑的场景。
